# 地图优先交易员驾驶舱规范 - CN

## 目的

Eurogas Nexus 首页是地图优先的欧洲天然气交易决策驾驶舱，不是通用仪表盘。它让交易员在一个主屏上理解上游资源池、推荐销售路径、容量与路线限制、市场变化、策略状态、预警和指示性 PnL。

首页必须是资源池原生设计。所有有效买入合同先被理解为组合资源池中的可用资源；地图和右侧决策栏展示这组资源应该如何销售，而不是只展示“当前选中的一份合同”。

## 首页布局

首屏必须以欧洲天然气地图为主。地图必须展示：

- 上游买入合同形成的资源池敞口；
- LNG 接收站、beach 交付点、hub、互联点、储气库、平衡区和主要路线走廊；
- 从资源位置到销售市场的候选路径；
- 可用路线、被阻断路线和拆分销售路径；
- TSO 准入约束；
- 后端有数据时的管容、流量、停运和费率状态；
- 与资源池相关的实时价格、价差和策略状态；
- 预警覆盖层。

资源标记必须指出资源池可用位置，例如 beach 交付点、LNG 接收站、hub、互联点、客户录入的交付点或 title-transfer 点。推荐销售路径必须同时可见，使交易员能看到整体分配、拆分路线和被阻断方案。

地图不能只是装饰背景，必须是主要工作界面。

## 首页信息架构

首页必须保持干净，只允许三个功能区：

1. 顶部栏：工作区菜单、全局搜索、运行状态、语言和主题。
2. 左侧栏：资源池上下文、推荐销售路径控制、当前组合资源。
3. 右侧栏：资源池 PnL、路线分配阶梯、经济性快照、策略/预警信号。

详细数据源诊断、PostgreSQL 健康状态、TSO 准入表、容量汇总、费率表和供应商凭据不属于首页，必须放在数据源中心、运行状态、市场、情景或合同页面。

如果后续重新加入紧凑实时行情条，它不能遮挡左右栏、MapLibre 控件或地图 attribution，并且只能显示：

- 资源池总量和未分配量；
- 相关 hub/day-ahead 参考价格；
- 有授权时的 ICE OCM 或其他日内标记；
- 涉及跨币种经济性时的 ECB FX；
- 实时指示性 PnL；
- 导入的外部屏幕订单姿态和组合 PnL 快照；
- 当前策略运行状态；
- 最新决策支持信号；
- 预警数量。

后端提供来源和新鲜度时，每个数值都必须显示来源和新鲜度。

## 地图交互

必须支持：

- 搜索终端、hub、TSO、平衡点、气田或数据源；
- 切换管网、LNG、互联点、hub、储气、天气、合同、市场、路线和预警图层；
- 点击路线或资产打开检查面板；
- 视觉化比较多条路线；
- 高亮阻断原因，例如缺少 TSO 准入、缺少费率、缺少管容、缺少接收站窗口、缺少供应商凭据或数据过期；
- 只通过后端 API 启动路线成本、LNG regas readiness、资源池优化或策略评估。

存在订单/PnL 上下文时，地图必须在相关节点之间画出高亮路线，并显示简洁标签和 PnL 数值。该动画仅用于决策支持和视觉理解，不代表提名、调度、订单路由或可执行交易路径。动画必须尊重 `prefers-reduced-motion`。

## 资源池决策栏

首页右侧栏必须总结组合决策：

- 资源池总预期 PnL；
- 已分配数量和未分配数量；
- 每条推荐销售路径的目标市场、分配数量、路线成本、销售价格、边际 netback、PnL 贡献、容量限制、TSO 准入状态和预警；
- 被阻断路线及明确阻断原因；
- 合同层 PnL 归因作为钻取信息，不作为首要决策。

如果最便宜路线容量有限，UI 必须显示该路线的部分分配，并显示剩余数量的下一最佳处理方式。若剩余数量绕路销售不经济，建议应显示本地销售、其他市场销售或保留/不销售，而不是强制走亏损路线。

## 合同录入

合同细节属于独立 Contracts 页面。页面必须遵循 `docs/contracts/21_RESOURCE_POOL_CONTRACT-CN.md` 的 EFET 风格结构：Agreement、Product and term、Delivery、Quantity/tolerance、Price、Costs、Capacity rights、Settlement/cash 和 Restrictions。首页只消费已存入 PostgreSQL 并通过 `/api` 暴露的资源池状态。

## 独立详情页面

详细工作不能挤占首页地图。必须提供独立页面：

- 价格监控；
- 组合和资源池；
- 合同录入；
- 路线成本和 LNG regas 分析；
- 策略回测、影子运行和实时监控；
- 导入的外部订单记录；
- 预警中心；
- 数据源血缘和授权状态；
- 术语库；
- 用户手册；
- 凭据和设置。

V1 中订单记录仅为导入/参考记录。产品不得下单、路由订单、修改订单、取消订单、捕获交易或提交提名。

## 策略 UX

策略页面必须支持回测、影子运行、实时监控、配置的资源池、交易窗口、bar 周期、SAP/ICIS/ICE OCM 对比、均值回归、评分、最佳时段、加权组合和风控限制。UI 必须使用决策支持语言，例如信号、候选、选项、目标、复核；不得使用执行性语言。

## 视觉质量标准

商业交付质量要求：

- 专业交易工作站式布局，信息密度高但可读；
- 克制配色和清晰语义状态；
- 不做营销首页、landing page 或装饰性填充；
- 稳定的响应式尺寸；
- 支持 light/dark/system 主题；
- 所有可见文字通过 i18n 支持英文和中文；
- 地图和数据面板在桌面与笔记本屏幕上可用；
- 移动端和平板降级为地图优先标签页，而不是压缩仪表盘。

## 数据边界

驾驶舱只能消费后端 `/api` 或 SDK 表面。不得直接读取 PostgreSQL、后端文件、供应商文件、`.env` 或凭据。

## 运行库数据缺失时的界面规则

首页可以显示简洁的运行库或输入缺失提示，但提示只能覆盖资源池优化所需的前置条件：PostgreSQL 运行库连接、已持久化的资源合同、候选路线、TSO 费率行和市场价格观测。首页不能变成数据源中心、运行状态页或费率表的重复页面。

资源池优化按钮必须在后端确认全部 PostgreSQL 输入齐备前保持不可用。客户端只能显示缺失项和下一步动作，并把用户引导到对应页面：数据源中心处理供应商和采集问题，运行状态页处理数据库就绪问题，合同页处理资源录入，情景页处理路线经济性，市场页处理价格观测。

如果数据缺失，客户端必须展示缺失状态。不得在浏览器侧生成 mock、sample、synthetic、generated 或 fallback 的路线、管容、费率、价格数据。演示或测试数据如有需要，必须写入 PostgreSQL，标注 demo provenance，再通过 `/api` 读取。
